# Atelier : rédiger des commits significatifs

**Rappel théorique** Un commit Git est bien plus qu'une simple sauvegarde : c'est une unité de travail documentée, traçable et communicable. Un bon message de commit répond à une question simple : « Que se passe-t-il dans ce commit et pourquoi ? » La convention la plus répandue repose sur un format en deux parties : une ligne de sujet courte (50 caractères max, à l'impératif, sans point final) et, si nécessaire, un corps explicatif séparé par une ligne vide. Les erreurs les plus fréquentes chez les débutants sont : des messages vagues comme 'fix', 'update', 'wip', des commits fourre-tout qui mélangent plusieurs modifications sans lien, ou à l'inverse des micro-commits inutiles pour une seule virgule corrigée. Un bon commit est atomique : il encapsule une seule intention cohérente. Exemple concret : - ❌ Mauvais : `git commit -m "modif"` - ✅ Bon : `git commit -m "Add email validation on registration form"` - ✅ Avec corps : message d'objet suivi d'une explication sur le pourquoi du changement. Retenir ce mental model : imagine que ton équipe devra relire l'historique dans 6 mois pour comprendre une décision. Chaque commit doit être une entrée de journal de bord claire et autonome.
