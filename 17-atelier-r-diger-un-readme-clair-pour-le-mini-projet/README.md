# Atelier : rédiger un README clair pour le mini-projet

**Rappel théorique** Un README est le premier fichier qu'un visiteur, recruteur ou collaborateur consulte lorsqu'il découvre un dépôt Git. C'est un fichier texte rédigé en Markdown (extension `.md`) qui sert de carte d'identité au projet : il explique ce que fait le projet, comment l'installer, comment l'utiliser et qui l'a créé. Un bon README ne se contente pas d'exister ; il doit être lisible en 30 secondes et permettre à quelqu'un d'extérieur de comprendre immédiatement la valeur et l'usage du projet. Les erreurs fréquentes des débutants : laisser le README vide ou avec le texte généré automatiquement par GitHub ('# mon-projet'), oublier les instructions d'installation, ne pas préciser les prérequis techniques, ou rédiger dans un style trop vague ('c'est un projet web'). À l'inverse, un README trop long et sans structure décourage la lecture. Mental model à retenir : imaginez que vous envoyez votre dépôt à un développeur junior qui ne vous connaît pas. Peut-il cloner le projet, l'installer et le lancer sans vous poser une seule question ? Si oui, votre README est réussi. Mini-exemple de structure minimale efficace : ``` # Nom du projet Courte description (1-2 phrases). ## Prérequis ## Installation ## Utilisation ## Auteur ``` Cette structure de base est un point de départ, pas une limite. --- **Énoncé de l'atelier** Vous avez réalisé un mini-projet de site statique versionné avec Git au cours des activités précédentes.
