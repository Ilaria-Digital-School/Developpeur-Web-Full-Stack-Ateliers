# Atelier : rendre un formulaire à la fois responsive et accessible

**Rappel théorique** Un formulaire accessible et responsive repose sur deux piliers complémentaires. L'accessibilité (a11y) garantit que tous les utilisateurs, y compris ceux utilisant des technologies d'assistance (lecteurs d'écran, navigation clavier), peuvent interagir avec le formulaire. Le responsive design assure une expérience cohérente sur tous les formats d'écran. Ces deux dimensions ne s'opposent pas : elles se renforcent mutuellement. Quelques définitions clés : l'attribut `for` sur un `` doit correspondre à l'`id` du champ associé — sans cela, le lecteur d'écran ne relie pas le label au champ. Les attributs `aria-required`, `aria-invalid` et `aria-describedby` enrichissent l'expérience des utilisateurs non-voyants. Le focus visible (outline) ne doit jamais être supprimé en CSS sans être remplacé. Le contraste minimum WCAG AA est de 4,5:1 pour le texte normal. Erreurs fréquentes : utiliser `placeholder` comme seul label (le placeholder disparaît à la saisie), supprimer `outline: none` sans alternative visuelle, empiler des champs sans penser à leur ordre de tabulation, ou oublier les messages d'erreur liés à leurs champs via `aria-describedby`. Mini-exemple : `Adresse e-mail Format invalide` — ce patron lie visuellement et sémantiquement le label, le champ et le message d'erreur. --- **Énoncé de l'atelier** Vous allez concevoir et coder un formulaire de contact complet, à la fois responsive (mobile-first) et accessible (conformité WCAG AA minimum).
