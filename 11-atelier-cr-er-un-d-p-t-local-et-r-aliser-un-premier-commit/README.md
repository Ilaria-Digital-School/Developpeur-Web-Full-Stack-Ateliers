# Atelier : créer un dépôt local et réaliser un premier commit

**Rappel théorique** Git est un système de contrôle de version distribué. Son rôle est de tracer l'historique des modifications de vos fichiers, de vous permettre de revenir en arrière et, à terme, de collaborer avec d'autres développeurs. Un dépôt (repository ou « repo ») est simplement un dossier surveillé par Git : il contient vos fichiers ainsi qu'un répertoire caché `.git` dans lequel Git stocke toutes les informations de versionnement. Le cycle de travail de base dans Git repose sur trois zones : le **répertoire de travail** (vos fichiers sur le disque), l'**index ou staging area** (zone de préparation où vous placez les modifications que vous voulez enregistrer) et le **dépôt local** (l'historique définitif, constitué de commits). Un commit est un instantané (snapshot) de vos fichiers à un moment précis, accompagné d'un message descriptif. La commande `git add` déplace les modifications vers le staging, et `git commit` les grave dans l'historique. Erreurs fréquentes à éviter : oublier de configurer `user.name` et `user.email` avant le premier commit (Git refusera ou utilisera des valeurs par défaut incorrectes), confondre `git init` (initialise un dépôt) avec `git clone` (copie un dépôt distant), et rédiger des messages de commit vides ou non descriptifs comme « test » ou « aaa ». Mini-exemple : vous créez un fichier `notes.txt`, vous tapez `git add notes.txt` pour le placer en staging, puis `git commit -m "Ajout du fichier notes"` pour l'enregistrer. Résultat : Git crée un premier commit avec un identifiant unique (hash SHA-1). --- **Énoncé de l'atelier**...
