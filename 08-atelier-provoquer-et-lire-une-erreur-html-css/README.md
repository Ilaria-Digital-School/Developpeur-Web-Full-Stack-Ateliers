# Atelier : provoquer et lire une erreur HTML/CSS

**Rappel théorique** Un développeur web passe une grande partie de son temps à lire et interpréter des erreurs. Ce n'est pas un signe d'échec, c'est une compétence fondamentale. En HTML et CSS, les erreurs ne bloquent généralement pas le rendu de la page comme en JavaScript, mais elles produisent des comportements inattendus : éléments mal affichés, styles non appliqués, structure cassée. Comprendre pourquoi quelque chose ne s'affiche pas comme prévu est un réflexe que tout développeur doit développer dès le départ. Les erreurs HTML les plus fréquentes sont : une balise non fermée (ex. `` sans ``), un attribut mal orthographié (ex. `clase` au lieu de `class`), une imbrication incorrecte de balises (ex. `texte`), ou un chemin de fichier CSS incorrect dans le ``. Côté CSS, les erreurs classiques incluent : une propriété inexistante, une valeur incorrecte (ex. `color: bleu;` au lieu de `color: blue;`), un sélecteur qui ne correspond à aucun élément, ou un fichier CSS qui n'est tout simplement pas chargé. La console et l'onglet Elements des DevTools du navigateur sont vos meilleurs alliés pour diagnostiquer ces problèmes. Un chemin CSS mal renseigné dans le `` produira une erreur réseau visible dans l'onglet Network. Un sélecteur CSS invalide sera simplement ignoré sans message d'erreur, ce qui rend le débogage visuel indispensable.
