# Atelier : nettoyer et commenter votre code

**Rappel théorique** Un code propre et bien commenté n'est pas un luxe réservé aux équipes senior : c'est une compétence professionnelle fondamentale dès le premier jour. Nettoyer son code (ou « refactoring léger ») consiste à supprimer tout ce qui est inutile (balises vides, espaces superflus, commentaires obsolètes, indentation incohérente), sans modifier le comportement visuel ou fonctionnel de la page. Un bon nommage de fichiers (minuscules, tirets, pas d'espaces) et une arborescence lisible facilitent la maintenance et la collaboration via Git. Les commentaires HTML s'écrivent avec la syntaxe et servent à expliquer l'intention d'un bloc, pas à paraphraser ce que le code dit déjà. Erreurs fréquentes : commenter chaque balise inutilement (bruit), laisser des blocs de code commenté « au cas où » sans raison, mélanger tabulations et espaces, ou imbriquer des éléments sans cohérence d'indentation. Mini-exemple : au-dessus d'un est utile ; ne l'est pas. Ce rappel fait le lien avec les ateliers précédents sur la sémantique, la hiérarchie des titres et les bonnes pratiques de nommage : désormais, vous allez appliquer tout cela sur un code réel à corriger. --- **Énoncé de l'atelier** Vous disposez d'un fichier HTML de départ volontairement mal écrit (fourni ci-dessous dans la section Instructions). Ce fichier représente la page d'accueil d'un blog fictif. Il contient de nombreuses imperfections : indentation chaotique, balises non sémantiques là où des balises appropriées existent, hiérarchie de titres incorrecte, commentaires absents ou inutiles, nommage de classes peu explicite, et du code mort (balises vides ou commentées sans justification)....
