# Atelier : corriger un bug d’affichage HTML/CSS avec DevTools

**Rappel théorique** Les DevTools (outils de développement) sont intégrés à tous les navigateurs modernes (Chrome, Firefox, Edge) et constituent l'outil de débogage de référence pour tout développeur web. On y accède via F12 ou clic droit > Inspecter. L'onglet 'Elements' (ou 'Inspecteur' sur Firefox) permet de visualiser le DOM en temps réel et les règles CSS appliquées à chaque élément. L'onglet 'Console' affiche les erreurs JavaScript et les logs. L'onglet 'Styles' (panneau droit dans Elements) liste toutes les règles CSS actives, héritées ou surchargées — les propriétés barrées indiquent qu'elles ont été écrasées par une règle plus spécifique. Le mental model à adopter face à un bug d'affichage : ne jamais modifier le code à l'aveugle. On observe d'abord (DevTools), on formule une hypothèse ("ce padding écrase ma mise en page"), on teste en direct dans le navigateur, puis seulement on applique la correction dans le code source. Cette méthode — observer, hypothèse, test, correction — est exactement la démarche de débogage pas à pas que vous avez vue dans le chapitre précédent. Erreurs fréquentes : oublier que les styles héritent en cascade, confondre margin et padding, ignorer le box-model (content + padding + border + margin), ne pas penser aux conflits de spécificité CSS. Mini-exemple : un élément div semble trop large ? Inspectez-le dans DevTools, regardez le box-model visuel en bas du panneau Styles, et cherchez un padding ou une border qui s'ajoute à la width déclarée. --- **Énoncé de l'atelier** Vous allez travailler sur une page HTML/CSS...
